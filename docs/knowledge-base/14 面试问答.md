## 一、React（5 题）

### Q1：你为什么选择 Taro + React 而不是原生微信小程序开发？

**回答**：
选择 Taro 主要基于三点：第一，跨端能力——Taro 支持编译到微信小程序、H5、React Native，未来扩展到其他平台不需要重写；第二，React 生态——可以使用 React Hooks（如 useEvaluate）、JSX 语法、TypeScript 类型检查，开发效率比原生 WXML/WXSS 高；第三，组件化——Taro 的组件系统与 React 一致，可以复用社区的组件库和设计模式。

**追问**：Taro 有什么缺点？

**追问回答**：
两个主要缺点。第一，编译差异——Taro 编译到小程序时，某些 React 特性不支持（如 dangerouslySetInnerHTML、Portal），需要查文档确认兼容性；第二，调试困难——编译后的代码是小程序原生代码，报错信息指向编译产物而非源码，定位问题需要 sourcemap 支持。

### Q2：useEvaluate Hook 的状态机设计是怎么考虑的？

**回答**：
useEvaluate 采用了有限状态机模式，定义了 4 个状态：idle（空闲）、loading（加载中）、success（成功）、error（失败）。状态转换是严格受控的：idle 只能转到 loading，loading 只能转到 success 或 error，error 只能转到 idle。这种设计防止了非法状态转换（比如从 error 直接跳到 success），也让 UI 渲染逻辑清晰——每个状态对应不同的 UI 表现。

**追问**：为什么不用 useState + if-else？

**追问回答**：
useState + if-else 在简单场景下没问题，但当状态和转换变多时，代码会变成一堆嵌套的 if-else，难以维护。状态机的优势是：状态和转换关系是显式的，可以在一个地方看到所有可能的状态和转换路径。这对于调试特别有用——如果 UI 显示异常，直接看当前状态就知道问题在哪。

### Q3：双层类型设计（AIResponse vs EvaluationResult）的考虑是什么？

**回答**：
AI 返回的数据用 snake_case（如 project_name、scores.resume.score），符合 JSON 的通用惯例，DeepSeek 的输出风格更稳定。前端使用 camelCase（如 projectName、scores.resume.score），符合 TypeScript/JavaScript 的命名惯例，IDE 补全更友好。中间通过 parser.ts 的 transformResponse 函数做转换，同时完成容错处理。这样前后端解耦，AI 输出格式的变化不会直接影响前端代码。

**追问**：为什么不直接在前端用 snake_case？

**追问回答**：
两个原因。第一，团队惯例——TypeScript 生态中 camelCase 是主流，使用 snake_case 会导致每次访问属性时都在"反直觉"；第二，IDE 支持——camelCase 的属性名在 IDE 补全时更自然，输入 `result.pro` 会提示 `projectName`，而 `result.pro` 不会提示 `project_name`。

### Q4：TailwindCSS 在小程序中的使用有什么坑？

**回答**：
两个主要坑。第一，单位问题——TailwindCSS 默认用 px，但小程序推荐用 rpx。需要在 tailwind.config.js 中配置单位转换；第二，类名编译——TailwindCSS 的 JIT 模式会扫描源码生成样式，但 Taro 的编译过程可能改变文件结构，导致某些类名没有被扫描到。解决方案是在 safelist 中配置常用类名。

**追问**：为什么不用 CSS Modules 或 styled-components？

**追问回答**：
TailwindCSS 的优势是快速原型——直接在 JSX 中写样式类名，不需要在 CSS 文件和 JS 文件之间切换。对于 Project Judge 这种"一个人开发、快速迭代"的项目，开发效率优先于代码组织。如果是团队项目或需要主题切换，CSS Modules 可能更合适。

### Q5：ScoreBar 组件是怎么设计的？

**回答**：
ScoreBar 是一个纯展示组件，接收 score（0-100）和 color 两个 props。内部用一个固定宽度的容器 + 一个动态宽度的填充条实现。宽度通过 `width: ${score}%` 动态计算。颜色通过 score.ts 的 getScoreColor 函数映射：>=80 绿色，>=60 蓝色，>=40 黄色，<40 红色。动画用 CSS transition 实现，加载时从 0 宽度过渡到目标宽度。

**追问**：为什么不用 Canvas 绘制？

**追问回答**：
CSS 方案更轻量、更易维护、支持动画过渡。Canvas 适合复杂图形（如海报生成），但对简单的进度条来说是过度工程。CSS 方案还有一个优势：可以在开发者工具中直接审查元素，调试方便。

## 二、TypeScript（3 题）

### Q6：AIResponse 和 EvaluationResult 两个接口的设计考虑是什么？

**回答**：
AIResponse 是 AI 返回的原始格式，字段名用 snake_case（与 Prompt Schema 对齐），类型定义尽量宽松（允许 undefined）。EvaluationResult 是前端使用的格式，字段名用 camelCase，类型定义严格（所有字段都有值，没有 undefined）。中间通过 transformResponse 做转换和默认值填充。这种设计的好处是：AI 输出格式变化只影响 parser.ts，不影响任何页面代码。

**追问**：为什么不直接用一个接口？

**追问回答**：
如果只用一个接口，要么前端用 snake_case（不规范），要么 Prompt 输出 camelCase（AI 不稳定）。两个接口虽然增加了维护成本，但提供了清晰的边界：AI 侧的变化不会传导到前端。

### Q7：EstimatedDevTime 类型为什么用字面量联合类型？

**回答**：
```typescript
type EstimatedDevTime = '1-3天' | '3-7天' | '1-2周' | '2-4周' | '1个月+'
```
用字面量联合类型而非 string，有两个好处：第一，类型安全——如果代码中拼错了值（如 `'1-3天'` 写成 `'1-3 天'`），TypeScript 会报错；第二，自文档化——看到这个类型就知道开发时间只有这 5 个可能的值，不需要查文档。

**追问**：如果未来要增加新的时间范围怎么办？

**追问回答**：
在 types/index.ts 中扩展联合类型的定义即可。由于 parser.ts 的 normalizeEstimatedDevTime 函数会将任何字符串映射到这 5 个值之一，所以即使 AI 输出了新的格式（如"大约 10 天"），也会被自动映射到最接近的枚举值。

### Q8：怎么处理 AI 返回的 any 类型？

**回答**：
AI 返回的数据经过 JSON.parse 后是 any 类型。我通过两层处理：第一层，parser.ts 中的每个解析函数都接收 unknown 类型（比 any 更安全），内部用类型守卫检查；第二层，transformResponse 函数返回 EvaluationResult 类型（严格类型），确保输出给页面的数据类型正确。这样从 AI 的"未知类型"到前端的"已知类型"，中间有一道类型安全的转换层。

## 三、云函数（3 题）

### Q9：微信云函数和自建服务器相比，优劣是什么？

**回答**：
微信云函数的优势：免运维、免鉴权（wx.cloud.callFunction 自动携带用户身份）、免域名备案、一键部署、按调用付费。劣势：只能在微信生态内使用、运行环境受限（Node.js 版本固定）、冷启动延迟（1-2 秒）、无法做长连接。对 Project Judge 来说，云函数是最优选择，因为：1）它是微信小程序，天然在微信生态内；2）请求-响应模式，不需要长连接；3）个人项目，运维成本是关键约束。

**追问**：冷启动怎么优化？

**追问回答**：
三个方案。第一，定时保活——用云开发的定时触发器，每 5 分钟发一个空请求，保持函数实例温热；第二，减少依赖——云函数只用 wx-server-sdk 和 node-fetch，减少包体积，加快加载；第三，前端预加载——在用户输入内容时就开始初始化云函数调用，减少感知延迟。

### Q10：云函数的错误处理是怎么做的？

**回答**：
云函数的错误处理分三层。第一层，输入校验——检查 content 是否为空、是否超长（5000 字）；第二层，GitHub API 错误——catch 后返回 null，评估继续（降级方案）；第三层，DeepSeek API 错误——区分超时（TIMEOUT）和其他错误，返回不同的错误信息。所有错误都通过 `{ success: false, error: '...' }` 格式返回给前端，前端根据 success 字段判断是否成功。

**追问**：为什么不抛异常让前端 catch？

**追问回答**：
微信云函数的异常会被框架捕获，返回给前端的是一个通用错误信息，丢失了自定义的错误详情。通过返回 `{ success: false, error }` 格式，前端可以拿到精确的错误信息，给用户更友好的提示。

### Q11：云函数的超时设置为什么是 60 秒？

**回答**：
60 秒的考虑：GitHub API 5 路并行调用约 1-2 秒，DeepSeek API 调用约 10-20 秒，加上网络波动和 Prompt 处理，正常情况 15-30 秒完成。60 秒留有 2 倍余量，覆盖极端情况（如 DeepSeek 高峰期响应慢）。微信云函数最大超时是 60 秒，已经用到了上限。

## 四、DeepSeek API（3 题）

### Q12：为什么选择 DeepSeek 而不是 GPT-4o？

**回答**：
三个原因。第一，成本——DeepSeek 的 API 单价约为 GPT-4o 的 1/10，对个人项目控制成本至关重要；第二，中文能力——DeepSeek 的中文场景 F1 评分接近 GPT-4o，Project Judge 的 Prompt 和输出都是中文；第三，JSON 支持——DeepSeek 原生支持 response_format=json_object，强制输出 JSON 格式，减少解析失败。

**追问**：如果 DeepSeek 涨价了怎么办？

**追问回答**：
架构上已经做了抽象——DeepSeek 的调用集中在 cloud/evaluate/index.js 的 callDeepSeek 函数中。如果需要切换到其他 LLM（如 GPT-4o、Claude），只需要修改这个函数的 API 地址和请求格式。Prompt 模板和 Parser 都不需要改动。

### Q13：temperature=0.3 是怎么确定的？

**回答**：
temperature 控制 AI 输出的随机性。0 = 完全确定性，0.7 = 较高随机性。0.3 是实验得出的平衡点：太低（0-0.1），AI 的评分过于保守，所有项目都像在套模板；太高（0.5-0.7），同一项目两次评估的分数差异超过 10 分，用户会质疑结果一致性。0.3 保证了"同一大方向下的合理波动"。

**追问**：有没有考虑用 top_p 替代 temperature？

**追问回答**：
top_p 和 temperature 是两种不同的采样策略。temperature 通过缩放 logits 控制随机性，top_p 通过截断概率分布控制随机性。DeepSeek 的文档建议不要同时使用两者。选择 temperature 是因为它的行为更直观——0 到 1 的范围直接对应"保守到激进"，而 top_p 的效果依赖于词表分布，不如 temperature 线性。

### Q14：response_format=json_object 和在 Prompt 中要求 JSON 有什么区别？

**回答**：
两者是互补的。response_format=json_object 是 DeepSeek 的原生功能，在 API 层面强制输出合法 JSON——即使模型想输出文本，也会被截断为 JSON。Prompt 中要求 JSON 是语义层面的约束——告诉模型"输出这个格式"。两者结合是双保险：response_format 保证格式合法，Prompt 保证内容符合 Schema。

## 五、Prompt 工程（5 题）

### Q15：你的 Prompt 经历了几个版本？每次改了什么？

**回答**：
经历了 6 个版本。V1：单分评估（score + confidence）；V2：六维评分（拆分维度）；V3：增加 Benchmark + 同类项目 + category；V4：增加面试题 + STAR 简历；V5：增加开发周期拆解；V6：最终版（20+ 字段，分层路由）。每次升级都是因为发现了新的用户需求或技术问题。

**追问**：最大的一次改动是什么？

**追问回答**：
V2 到 V3 的改动最大。从"只有评分"到"评分 + Benchmark + 同类项目 + 分类"，Prompt 长度翻了一倍。关键是引入了"AI 分类 + 规则校验"的混合模式——AI 负责判断项目类别，但 Benchmark 计算用真实数据。这个设计解决了"分数没有参考系"的根本问题。

### Q16：分层路由策略是怎么设计的？

**回答**：
根据输入类型选择不同的 Prompt intro。GitHub + 有 repoInfo：强调"基于真实数据评估"，注入仓库元数据；GitHub + 无 repoInfo：通用开源评估；description：综合评估；idea：强调可行性分析。核心思想是：同一个 AI 角色，根据信息量不同，给出不同深度的评估。信息越充分，评估越准确。

**追问**：为什么不给每种类型设计完全不同的 Prompt？

**追问回答**：
因为输出格式是统一的（20+ 字段的 JSON），只有 intro 不同。如果每种类型设计完全不同的 Prompt，维护成本会翻 4 倍，而且输出格式不一致会导致 parser 需要处理 4 种不同的结构。

### Q17：怎么保证 AI 不编造 similar_projects？

**回答**：
在 Prompt 中明确写了"similar_projects 必须是真实存在的项目，不要编造"。但这只是"软约束"——AI 仍可能编造看似合理但不存在的项目。目前的缓解策略是：1）GitHub 项目评估时，注入真实的 README 和技术栈，AI 更容易推荐相关的真实项目；2）前端展示时，similar_projects 的名称会高亮显示，用户可以自行搜索验证。未来可能接入搜索 API 做自动验证。

### Q18：Prompt 中的示例格式有什么作用？

**回答**：
示例格式比抽象规则更有效。比如要求"answer 必须有具体数据"，AI 可能给出"使用了先进的技术"这种空话。但如果给一个示例："选择 DeepSeek 主要基于三点：第一，API 单价仅为 GPT-4o 的约 1/10……"，AI 会自动模仿这个结构：分点论述 + 数据支撑 + 技术术语。这是 Prompt 工程中最实用的技巧之一。

### Q19：20+ 个字段的 Prompt 会不会太长？

**回答**：
会，这是当前架构的一个权衡。Prompt 约 2000 字（中文），加上用户输入和 GitHub 数据，单次请求的 input token 约 3000-5000。DeepSeek 的上下文窗口是 32K，远未触及上限。但更长的 Prompt 意味着更高的 token 费用和 AI 更容易遗漏字段。缓解策略：字段字数限制（减少 output token）、response_format 强制 JSON（减少格式错误）、parser 容错（兜底处理）。

## 六、Benchmark Engine（3 题）

### Q20：Benchmark Engine 的设计思路是什么？

**回答**：
核心问题是"AI 的评分没有参考系"。解决方案是建立 15 个项目类别的基准数据（benchmark.json），每个类别有 avg/top30/top20/top10 四个百分位阈值。AI 输出 recommendation.score 和 category 后，规则引擎计算百分位排名。关键设计是"AI 分类 + 规则计算"——不信任 AI 给出的百分位数字，用真实数据覆盖。

**追问**：基准数据从哪来？

**追问回答**：
当前的基准数据是基于行业经验估计的，不是真实统计。例如 agent 类别的 avg=75，是因为 2026 年 AI Agent 是热门赛道，整体质量较高。未来计划接入真实数据源（如 GitHub Star 分布、Product Hunt 评分分布）来校准。

### Q21：为什么只对 recommendation 做 Benchmark，其他维度不做？

**回答**：
两个原因。第一，recommendation 是综合维度，已隐含了其他维度的信息；第二，如果对 6 个维度都做 Benchmark，基准数据量从 60 个膨胀到 360 个，维护成本显著增加。这是一个 MVP 权衡。

### Q22：为什么用规则引擎计算 Benchmark 而不是让 AI 直接输出？

**回答**：
因为 AI 的 benchmark 输出不可靠。实验发现，AI 给出的百分位数字波动很大——同一个项目，一次说 Top 20%，一次说 Top 35%。而规则引擎基于固定的阈值计算，结果完全确定。所以设计了"AI 负责分类，规则引擎负责计算"的混合模式。

## 七、GitHub API（3 题）

### Q23：5 路并行 API 调用的设计是怎么考虑的？

**回答**：
需要获取仓库的完整画像：元数据、README、技术栈、贡献者数、发布数。如果串行调用，总耗时约 5 秒（每个 API 1 秒）。用 Promise.all 并行调用，总耗时取决于最慢的那个，约 1-2 秒。这是一个典型的 IO 密集型优化——多个独立的网络请求，并行是最优解。

**追问**：如果某个 API 失败了怎么办？

**追问回答**：
整个 fetchGitHubRepo 函数被 try-catch 包裹。任何一个 API 失败（或全部失败），都返回 null。前端拿到 null 后，评估仍然继续，只是没有 GitHub 真实数据。这是一个"降级方案"——GitHub 分析是加分项，不是必须项。

### Q24：贡献者数量是怎么巧妙获取的？

**回答**：
GitHub 的 contributors API 默认返回 30 个贡献者。要获取精确总数，通常需要请求所有页。但有一个技巧：只请求第 1 页（per_page=1），从 Link header 的 rel="last" page 值推算总数。当 per_page=1 时，总页数 = 总贡献者数。这样只需要一次请求。

### Q25：技术栈提取为什么只识别 40+ 个已知框架？

**回答**：
从 package.json 的 dependencies 中提取技术栈，如果不过滤，会把所有依赖都列出来（包括内部包、工具包、类型定义包）。用户不需要知道项目用了 @types/node 或 lodash 这种基础库。只识别 40+ 个主流框架和库（React、Vue、Express、Prisma 等），让技术栈列表更有参考价值。

## 八、AI 产品设计（4 题）

### Q26：AI 产品的"信任边界"是什么？

**回答**：
核心认知：AI 的输出是"参考"，不是"事实"。在 Project Judge 中，我通过四层验证保证输出可靠性：1）response_format 强制 JSON 格式；2）Prompt 约束字段内容；3）Benchmark 用真实数据覆盖 AI 的 benchmark；4）Parser 用 11+ 个容错函数处理异常输出。永远不要让 AI 的输出直接呈现给用户，中间必须有"验证层"。

### Q27：删除 confidence 字段的决策过程是什么？

**回答**：
confidence 字段的设计意图是告诉用户"AI 对评分有多大把握"。但实际测试发现：同一项目两次评估，confidence 波动超过 20%；AI 对错误评估也能给出 confidence=0.95。根本原因是 LLM 的 confidence 不等于统计学的置信区间，它是"语气的确定程度"。解决方案：删除 confidence，用 Benchmark 百分位替代——用客观数据告诉用户"你排第几"，而非让 AI 自我评估。

**追问**：有没有做过 A/B 测试？

**追问回答**：
没有正式的 A/B 测试，因为是个人项目，用户量不足以支撑统计显著性。但通过用户反馈做了定性对比：保留 confidence 时，用户反馈"看到 confidence 忽高忽低，更不信任结果"；替换为 Benchmark 后，用户反馈"看到 Top 30%，比一个数字更有参考价值"。

### Q28：功能冻结的标准是什么？

**回答**：
三个条件同时满足时冻结：1）核心决策链路闭环——评估、评分、参考、行动建议、面试准备、分享，缺一不可；2）Prompt 字段数接近上限——20+ 字段已经让 AI 输出不稳定，继续增加会恶化；3）维护成本超过新功能价值——每增加一个功能，parser + types + Prompt 都要同步更新。v1.5 满足这三个条件，冻结。

### Q29：如果要商业化，你会怎么改？

**回答**：
三个方向。第一，数据层——从本地存储迁移到云端数据库，支持跨设备同步和多用户；第二，付费模式——基础评估免费，深度分析（GitHub 深度分析、面试题、STAR 简历）付费；第三，企业版——支持团队协作、项目库管理、批量评估报告导出。技术上需要：增加用户系统、接入支付、将云函数迁移到独立后端（支持更高并发）。

## 九、项目管理（3 题）

### Q30：6 周开发周期是怎么分配的？

**回答**：
Week 1：MVP 原型（前端直调 DeepSeek，单分评估）；Week 2：架构迁移（切换到云函数，六维评分）；Week 3：GitHub 集成（5 路并行 API，技术栈提取）；Week 4：功能扩展（Benchmark、面试题、STAR 简历）；Week 5：交互优化（批量对比、海报生成、历史记录）；Week 6：测试 + 修复 + 功能冻结。每个阶段都有可交付的产物。

### Q31：一个人开发，怎么做 Code Review？

**回答**：
让 Claude 做 Code Review。具体做法：写完一个模块后，把代码贴给 Claude，让它从安全性、可维护性、类型安全三个角度审查。Claude 发现了 extractJson 的边界问题、Benchmark 计算的时序问题、dead code 等问题。虽然 AI 的 Code Review 不能替代人工审查，但对个人项目来说，比不做 Review 好得多。

**追问**：Claude 的 Review 有没有漏掉什么？

**追问回答**：
有。effort.ts 的 dead code 是我自己发现的，Claude 没有主动提示删除。另外，contract-test.ts 中的 confidence 和 suitable_for 字段残留，Claude 也没有提醒更新测试用例。这说明 AI 的 Review 更擅长发现"当前代码的问题"，不太擅长发现"代码与文档/测试的不一致"。

### Q32：如果团队扩大到 3 人，你会怎么分工？

**回答**：
1 人负责前端（React + Taro + UI/UX），1 人负责后端/AI（云函数 + Prompt + Benchmark），1 人负责数据/测试（GitHub API + 测试用例 + 数据分析）。关键的协作点是 types/index.ts——它是前后端的契约，任何字段变更都需要三方同步。建议用 Git + PR 流程，types 变更必须三方 review。

### Q33：开发过程中最大的技术风险是什么？

**回答**：
最大的技术风险是 AI 输出的不确定性。传统软件的输入输出是确定的——给定输入，输出可预测。但 AI 的输出每次不同，甚至可能格式错误、字段缺失、内容编造。这导致：1）测试困难——无法用固定输入预期固定输出；2）UI 设计困难——字段可能为空、数字可能异常；3）用户信任困难——同一项目两次评估分数不同。缓解策略是"不信任 AI"——Parser 容错、Benchmark 覆盖、verdict 规则化。

### Q34：你从这个项目中学到的最重要的一课是什么？

**回答**：
最重要的一课是：**AI 产品的核心竞争力不是 AI 本身，而是围绕 AI 构建的"可靠性工程"**。DeepSeek 的能力是标准化的——任何人都可以调用同一个 API。但如何让 AI 的输出稳定可靠（Parser 容错）、有参考价值（Benchmark）、可直接使用（STAR 简历、面试题），这些"围绕 AI 的工程"才是产品的差异化。
