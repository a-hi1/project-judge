# 架构设计

> 从前端直调 DeepSeek 到云函数中转的完整架构演化

---

## 背景

Project Judge 的架构经历了 **一次重大转折**：从"前端直接调用 DeepSeek API"改为"前端 → 云函数 → DeepSeek"。这个转折发生在项目早期（v0.1 → v0.3），但其影响贯穿了整个项目生命周期。

## 最初方案：前端直调 DeepSeek

### 架构图

```
┌─────────────┐     HTTPS      ┌─────────────────┐
│  微信小程序   │ ──────────────→ │  DeepSeek API    │
│  (前端)      │ ←────────────── │  (第三方服务)     │
└─────────────┘    JSON 响应     └─────────────────┘
```

### 实现方式

```typescript
// 早期方案：前端直接 fetch
const response = await fetch('https://api.deepseek.com/chat/completions', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${API_KEY}`,  // ← 致命问题
  },
  body: JSON.stringify({
    model: 'deepseek-chat',
    messages: [...],
  }),
})
```

### 为什么放弃

**致命问题：API Key 暴露**

微信小程序的代码是**打包后下发到用户手机**的。即使做了代码混淆，通过抓包或反编译，任何人都能提取到 API Key。一旦 Key 泄露：
- 他人可以无限制调用 DeepSeek API，费用由你承担
- DeepSeek 可能因异常调用封禁你的 Key
- 无法控制调用频率和用量

**次要问题**：
1. **微信小程序域名白名单**：需要在微信后台配置 `api.deepseek.com` 为合法域名，增加配置步骤
2. **无法做服务端逻辑**：GitHub API 调用、Benchmark 计算、Prompt 模板组装等逻辑全部要放在前端，增加包体积
3. **无法做请求鉴权**：任何人都可以直接调用你的接口，无法区分合法用户和恶意请求
4. **超时控制困难**：小程序端的请求超时处理不如服务端灵活

### 错误决策的本质

这不是"技术选型错误"，而是"安全意识不足"。在项目最初的快速原型阶段，为了"尽快看到效果"，选择了最简单的实现方式。这个决策在 demo 阶段没问题，但在准备上线时暴露了致命缺陷。

**教训**：涉及 API Key 的场景，**Day 1 就应该用服务端中转**。这不是可以"后面再改"的事情——架构改动的成本会随着代码量指数级增长。

## 最终方案：云函数中转

### 架构图

```
┌─────────────┐   wx.cloud.callFunction   ┌──────────────────┐
│  微信小程序   │ ─────────────────────────→ │  云函数 evaluate   │
│  (前端)      │ ←───────────────────────── │  (Node.js)        │
└─────────────┘      { success, data }     └────────┬─────────┘
                                                     │
                                          ┌──────────┴──────────┐
                                          │                     │
                                     ┌────▼─────┐        ┌─────▼──────┐
                                     │ DeepSeek  │        │ GitHub API │
                                     │   API     │        │ (5路并行)  │
                                     └──────────┘        └────────────┘
```

### 为什么选择云函数

**备选方案 A：自建服务器（Express / Koa）**
- 优点：完全可控，可部署在任何云厂商
- 缺点：需要运维、需要域名备案、需要 SSL 证书、需要自己做鉴权
- 结论：对个人项目来说运维成本过高

**备选方案 B：Serverless 云函数（腾讯云 SCF / 阿里云 FC）**
- 优点：免运维、按调用付费
- 缺点：冷启动问题、需要额外配置 API 网关
- 结论：可行，但多了一层配置

**备选方案 C：微信云函数**
- 优点：天然集成微信生态、免鉴权（`wx.cloud.callFunction` 自动携带用户身份）、无需域名备案、一键部署
- 缺点：只能在微信生态内使用、运行环境受限（Node.js）、无法做长连接
- 结论：**最优选择**。Project Judge 是微信小程序，用云函数是最自然的方案

**最终决策**：方案 C（微信云函数）。

### 云函数核心设计

**文件位置**：`cloud/evaluate/index.js`

**职责**：
1. 接收前端传入的 `content`（项目内容）和 `type`（输入类型）
2. 如果是 GitHub 项目，调用 GitHub API 获取仓库信息
3. 组装 Prompt 模板（根据 type 选择不同模板）
4. 调用 DeepSeek API 获取评估结果
5. 用 Benchmark 数据覆盖 AI 生成的 benchmark 字段
6. 返回结构化结果

**关键设计决策**：

```javascript
// 温度参数：0.3
// 为什么不是 0？为什么不是 0.7？
// 0.3 = 平衡"确定性"和"创造性"
// 太低（0）：评分过于保守，所有项目都像在套模板
// 太高（0.7）：评分波动大，同一项目两次评估差异明显
temperature: 0.3

// 响应格式：强制 JSON
// 这是 DeepSeek 的原生能力，比"请用 JSON 格式输出"的 Prompt 约束更可靠
response_format: { type: 'json_object' }

// 超时：60 秒
// GitHub API（5路并行）+ DeepSeek 调用，正常情况 15-30 秒
// 60 秒留有余量，避免网络波动导致超时
TIMEOUT_MS = 60000
```

### 数据流详解

```
用户输入 "https://github.com/owner/repo"
  │
  ▼
前端 detectInputType() → "github"
  │
  ▼

  │
  ▼
云函数入口 main()
  │
  ├─→ fetchGitHubRepo(url)
  │     ├─→ GET /repos/:owner/:repo          (元数据)
  │     ├─→ GET /repos/:owner/:repo/readme   (README)
  │     ├─→ GET /repos/:owner/:repo/contents/package.json (技术栈)
  │     ├─→ GET /repos/:owner/:repo/contributors?per_page=1 (贡献者数)
  │     └─→ GET /repos/:owner/:repo/releases?per_page=1     (发布数)

        │
        ▼
页面渲染 EvaluationResult
```

## 前端架构

### 状态管理：useEvaluate Hook

**文件**：`src/hooks/useEvaluate.ts`

采用 **状态机模式**管理评估流程：

```
idle → loading → success
                → error → idle
```

**为什么用状态机而不是 useState？**
- 评估流程有明确的状态转换：空闲 → 加载中 → 成功/失败
- 状态机可以防止非法转换（比如从 error 直接跳到 success）
- 每个状态对应不同的 UI 表现，状态机让这个映射关系清晰

### 类型系统：双层类型设计

**文件**：`src/types/index.ts`

```typescript
// AI 返回的原始格式（snake_case，与 Prompt Schema 对齐）
interface AIResponse {
  project_name: string
  scores: { resume: { score: number; reason: string } }
  // ...
}

// 前端使用的格式（camelCase，符合 TS 惯例）
interface EvaluationResult {
  projectName: string
  scores: { resume: { score: number; reason: string } }
  // ...
}
```

**为什么设计两层类型？**
1. **AI 输出用 snake_case**：这是 JSON 的通用惯例，DeepSeek 的输出风格更稳定
2. **前端用 camelCase**：TypeScript/JavaScript 的命名惯例，IDE 补全更友好
3. **中间层 parser.ts 做转换**：`transformResponse` 函数同时完成 snake_case → camelCase 和容错处理

**备选方案**：直接在前端用 snake_case
- 优点：省去转换层
- 缺点：与 TypeScript 生态惯例冲突，每次访问都要写 `result.project_name` 而不是 `result.projectName`

### 设计系统

**文件**：`src/utils/theme.ts`

灵感来源：**Linear / Raycast / Vercel Dashboard** 的暗色主题。

核心设计原则：
1. **三层背景色**：`#0A0A0A`（bg）→ `#111111`（card）→ `#1A1A1A`（elevated），通过明度差异建立视觉层级
2. **8rpx 基础间距**：所有间距都是 8 的倍数（8/16/24/32/48/64），保证视觉一致性
3. **语义化颜色**：success（绿）、danger（红）、warning（黄）+ 对应的半透明背景色
4. **复用样式对象**：`cardStyle`、`pillStyle`、`dividerStyle`，避免重复代码

## 优缺点分析

| 优点 | 缺点 |
|------|------|
| API Key 安全，永不暴露到前端 | 云函数冷启动可能增加 1-2 秒延迟 |
| 免鉴权，`wx.cloud.callFunction` 自动携带身份 | 微信生态绑定，无法复用到其他平台 |
| 服务端可以做复杂逻辑（GitHub API、Benchmark） | 云函数运行环境受限，某些 npm 包不可用 |
| 双层类型设计保证了前后端解耦 | parser.ts 维护成本随字段增加而增长 |
| 状态机模式防止了非法状态转换 | 没有全局状态管理（如 Redux），跨页面状态传递靠 props |

## 面试官可能追问

1. "为什么不直接用 OpenAI 的 API？"
2. "云函数冷启动怎么优化？"
3. "如果并发量增大 10 倍，这个架构还成立吗？"
4. "为什么不把 Benchmark 计算放在前端？"
5. "双层类型设计会不会导致维护两份类型定义的负担？"

## 标准回答

> **Q：如果并发量增大 10 倍，这个架构还成立吗？**
>
> 短期内成立，但有天花板。微信云函数的单实例并发是 1，10 倍并发意味着需要 10 个实例同时运行。云函数会自动扩容，但有两个瓶颈：第一，DeepSeek API 的 rate limit（免费版约 60 RPM），需要申请提额或做请求队列；第二，GitHub API 未认证限制是 60 次/小时/IP，云函数出口 IP 共享可能触发限制。长期方案是：1）申请 GitHub Token 提升到 5000 次/小时；2）引入 Redis 缓存 GitHub 数据（同一仓库 24 小时内不重复请求）；3）DeepSeek API 升级到付费版提升 rate limit。
