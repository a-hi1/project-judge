# 面试题生成模块

> 预判面试官的问题，帮用户准备好答案

---

## 背景

做项目的人中，超过 60% 是为了求职。面试官一定会问"你做了什么项目？怎么做的？"。Project Judge 自动为每个项目生成 5 个面试题，每个题包含标准回答、追问、难度等级和关键词。

## 设计目标

1. 生成 5 个高质量面试题，覆盖项目的核心技术点
2. 每个问题有标准回答（不是空话，是像真实校招回答一样）
3. 每个问题有追问（面试官会追问的更深层问题）
4. 难度分级（1-5），让用户知道哪些需要重点准备
5. 关键词标记，便于快速定位技术要点

## 实现细节

### Prompt 设计

```json
{
  "interview_questions": [
    {
      "question": "面试官可能问的问题",
      "answer": "标准回答，100字以内，必须包含具体数据或技术细节",
      "follow_up": "面试官的追问，20字以内",
      "difficulty": 3,
      "keywords": ["关键词1", "关键词2"],
      "category": "架构设计 或 Prompt工程 或 性能优化 或 容错设计 或 技术选型"
    }
  ]
}
```

### 五个维度覆盖

Prompt 明确要求 5 个问题必须覆盖以下 5 个维度：

| 维度 | 示例问题 | 考察能力 |
|------|---------|---------|
| 技术选型 | "为什么选择 DeepSeek 而不是 GPT-4o？" | 技术判断力 |
| 架构设计 | "为什么用云函数而不是自建服务器？" | 架构思维 |
| Prompt 工程 | "你的 Prompt 经历了几个版本？每次改了什么？" | AI 应用能力 |
| 性能优化 | "如果并发量增大 10 倍，怎么优化？" | 工程能力 |
| 容错设计 | "AI 返回的 JSON 格式不对怎么办？" | 健壮性设计 |

### answer 的质量约束

这是整个模块最关键的设计：

```
answer 必须像真实的校招面试回答，有具体技术细节，禁止空话套话
answer 示例格式："选择 DeepSeek 主要基于三点：第一，API 单价仅为 GPT-4o 的约 1/10，
适合个人项目控制成本；第二，中文场景下 F1 评分接近；第三，支持 response_format=json_object，
输出稳定性高。"
```

**设计技巧**：
1. **"三点"结构**：分点论述是面试回答的黄金格式，面试官喜欢听"第一、第二、第三"
2. **数据支撑**：每个点都有具体数据或技术细节
3. **示例比规则更有效**：给 AI 一个高质量示例，它会自动模仿

### follow_up 的设计

```
follow_up 是面试官听了你的回答后会追问的更深层问题，
例如"如果并发量增大10倍，这个方案还成立吗？"
```

**为什么需要追问？** 因为面试官不会只问一个问题。他们会根据你的回答层层深入：
1. "你为什么用 X 技术？"（表面）
2. "如果条件变了怎么办？"（深入）
3. "还有其他方案吗？"（更深入）

follow_up 让用户提前准备好第二层回答。

### difficulty 分级

| 等级 | 含义 | 准备建议 |
|------|------|---------|
| 1-2 | 基础题 | 了解即可 |
| 3 | 中等题 | 需要准备具体回答 |
| 4 | 进阶题 | 需要深入理解技术细节 |
| 5 | 挑战题 | 需要有自己的思考和见解 |

### category 分类

5 个类别确保覆盖面：
- **架构设计**：系统设计、技术选型、模块划分
- **Prompt 工程**：AI 应用、Prompt 设计、输出控制
- **性能优化**：并发、缓存、延迟
- **容错设计**：异常处理、降级策略、数据校验
- **技术选型**：为什么用 A 而不用 B

## 前端解析

**文件**：`src/services/parser.ts:84-97`

```typescript
function parseInterviewQuestions(val: unknown): InterviewQuestion[] {
  if (!Array.isArray(val)) return []
  return val
    .filter((item): item is Record<string, unknown> => !!item && typeof item === 'object')
    .map(item => ({
      question: ensureString(item.question, '未知问题'),
      answer: ensureString(item.answer, '暂无参考答案'),
      follow_up: ensureString(item.follow_up, ''),
      difficulty: typeof item.difficulty === 'number' ? Math.max(1, Math.min(5, Math.round(item.difficulty))) : 3,
      keywords: ensureStringArray(item.keywords),
      category: ensureString(item.category, '技术选型'),
    }))
    .slice(0, 5)
}
```

**容错设计**：
- `difficulty` 默认为 3（中等）
- `follow_up` 默认为空字符串
- `category` 默认为"技术选型"
- 最多保留 5 个问题（`.slice(0, 5)`）

## 优缺点分析

| 优点 | 缺点 |
|------|------|
| 5 维度覆盖，不遗漏核心技术点 | AI 生成的回答可能过于模板化 |
| follow_up 让用户准备第二层回答 | 不同项目的面试题可能雷同 |
| difficulty 分级便于优先级排序 | AI 对难度的判断可能不准 |
| 示例格式保证回答质量 | 100 字限制可能限制深度 |

## 面试官可能追问

1. "5 个问题够吗？会不会遗漏重要知识点？"
2. "AI 生成的面试回答，面试官会不会觉得是背的？"
3. "怎么保证不同项目生成的面试题不雷同？"

## 标准回答

> **Q：AI 生成的面试回答，面试官会不会觉得是背的？**
>
> 可能会，如果用户逐字背诵的话。但设计意图不是让用户背答案，而是帮他们建立回答的框架。比如"选择 DeepSeek 主要基于三点"这个结构，用户可以用自己的话重新组织。面试官考察的是思维过程，不是记忆力。所以前端展示时，我们把 follow_up 放在答案下面（需要点击展开），引导用户理解"为什么这样回答"，而不是死记硬背。
