# 产品设计演化

> 从"AI 评分工具"到"AI 项目决策助手"的完整演化路径

---

## 背景

Project Judge 的产品定位经历了 **3 次重大转折**，每一次转折都源于用户反馈或实际使用中发现的问题。这不是一个预先规划好的产品路线，而是一个在实战中不断修正方向的真实案例。

## 第一阶段：AI 评分工具（v0.1 - v0.3）

### 初始设想

最早的产品定义非常简单：**用户输入一个项目想法，AI 返回一个 0-100 的评分**。

核心假设：
- 用户需要一个"分数"来判断项目是否值得做
- AI 能够给出可靠的量化评估
- 一个总分就够了

### 为什么这样设计

当时的核心认知是：**量化 = 决策依据**。一个数字就能告诉用户"做还是不做"。

Prompt V1 的输出非常精简：
```json
{
  "project_name": "...",
  "score": 78,
  "confidence": 0.85,
  "verdict": "recommend",
  "pros": [...],
  "cons": [...]
}
```

关键字段：`score`（总分）和 `confidence`（置信度）。

### 错误决策 #1：confidence 字段

**设计意图**：告诉用户"AI 对这个评分有多大把握"。比如 confidence = 0.85 意味着 AI 有 85% 的把握这个评分是准确的。

**实际问题**：
1. **AI 对 confidence 的理解极不稳定**。同一个项目输入两次，一次 confidence = 0.85，一次 confidence = 0.62。用户看到后会困惑："到底哪个是对的？"
2. **confidence 和实际准确度没有相关性**。AI 可以对一个完全错误的评估给出 confidence = 0.95，因为它"觉得自己很确定"。
3. **用户不知道怎么用这个数字**。confidence = 0.73 是高还是低？该不该信任？

**错误本质**：把 AI 的"自我感知"当成了客观指标。LLM 没有真正的校准能力（calibration），它的 confidence 更多是"语气的确定程度"，而非"统计意义上的置信区间"。

**修正方案**：删除 confidence 字段。详见下方"为什么删除 confidence"。

### 错误决策 #2：单一总分

**设计意图**：一个数字定生死。78 分 vs 65 分，高下立判。

**实际问题**：
1. **维度缺失**。一个"简历价值 90 分但商业价值 30 分"的项目，总分可能是 60——看起来平庸，但对求职者来说是极好的选择。
2. **用户画像不同**。有人为了求职，有人为了学习，有人为了赚钱。同一个总分对不同人意味着完全不同的决策。
3. **无法指导行动**。总分 65 分，然后呢？用户不知道该从哪里改进。

**修正方案**：拆分为多维度评分。详见下方"六维评分的诞生"。

## 第二阶段：多维评估工具（v0.5 - v0.8）

### 六维评分的诞生

经过 v0.1-v0.3 的教训，产品方向从"一个总分"变为"六个维度"：

| 维度 | 含义 | 分数越高代表 |
|------|------|-------------|
| resume | 简历价值 | 越适合写进简历 |
| learning | 学习价值 | 能学到越多新技术 |
| business | 商业价值 | 越有变现/市场需求 |
| difficulty | 技术难度 | 越难（100 = 极难） |
| innovation | 创新程度 | 越新颖，竞品越少 |
| recommendation | 综合推荐 | 最终推荐程度 |

### 为什么这样设计

**备选方案 A**：三维度（简历 / 学习 / 商业）
- 优点：简单，用户易理解
- 缺点：缺少"难度"和"创新"两个关键决策维度
- 结论：不够用

**备选方案 B**：八维度（加上用户体验、市场时机、团队匹配）
- 优点：更全面
- 缺点：信息过载，AI 打分更不稳定
- 结论：过度设计

**最终决策**：六维度。覆盖了求职者最关心的所有决策维度，同时不超出 AI 的稳定输出能力。

### 为什么增加 benchmark

**问题**：六维评分解决了"维度"问题，但没有解决"参考系"问题。resume = 75 分，这意味着什么？比平均好？还是比平均差？

**设计思路**：引入 Benchmark Engine，让每个分数都有参照物。

具体方案：
- 建立 15 个项目类别的基准数据（benchmark.json）
- 每个类别有 avg / top30 / top20 / top10 四个百分位阈值
- AI 输出 recommendation.score 后，用真实基准数据计算百分位
- 用户看到的不再是"75 分"，而是"75 分 = Top 30%，高于该类别平均水平（68 分）"

详见 [[05 Benchmark Engine设计]]。

### 为什么增加 similar_projects

**问题**：用户评估一个项目时，最常问的下一个问题是"有没有类似的？"

**设计思路**：AI 自动推荐 2-3 个真实存在的同类项目，并说明：
- 相似度（0-100）
- 一句话介绍
- 成功原因

**关键约束**：similar_projects 必须是真实存在的项目，不允许编造。Prompt 中明确写了 `"不要编造数据"`。

### 为什么增加 interview_questions

**洞察**：做项目的人中，很大比例是为了求职。面试官一定会问"你做了什么项目？怎么做的？"

**设计思路**：预判面试官的问题，帮用户准备好答案。每个问题包含：
- question：面试官的问题
- answer：标准回答（必须有具体数据和技术细节）
- follow_up：面试官会追问的更深层问题
- difficulty：难度等级（1-5）
- keywords：关键词
- category：分类（架构设计 / Prompt 工程 / 性能优化 / 容错设计 / 技术选型）

详见 [[08 面试题生成模块]]。

## 第三阶段：AI 项目决策助手（v1.0 - v1.5）

### 从"评估"到"决策"

v0.8 时，产品已经有六维评分 + Benchmark + 面试题 + STAR 简历。但用户反馈揭示了一个核心问题：**评估报告太长，用户不知道该看什么**。

**洞察**：用户要的不是"评估报告"，而是"做还是不做"的决策。

**修正方向**：
1. 增加 `verdict` 字段（recommend / not_recommend），基于 recommendation.score >= 60 自动判定
2. 增加 `one_line_conclusion`，一句话结论
3. 增加 `estimated_dev_time`，告诉用户"做这个要多久"
4. 增加 `development_breakdown`，把工期拆成具体任务
5. 增加 `mvp_suggestion`，给出最小可行方案

至此，产品从"评分工具"正式演化为"项目决策助手"。

### 为什么增加 compare

**场景**：用户手里有 3 个项目想法，想选一个最值得做的。

**备选方案 A**：让用户逐个评估，自己对比
- 优点：实现简单
- 缺点：用户要反复切换页面，心理负担大

**备选方案 B**：一次性输入多个项目，自动排序
- 优点：决策效率极高
- 缺点：实现复杂度增加（顺序调用、进度展示、排序高亮）

**最终决策**：方案 B。最多支持 5 个项目，逐个调用云函数，完成后按 recommendation.score 降序排列，最高分高亮。

详见 [[14 面试问答]] Q15（批量对比的设计挑战）。

### 为什么删除 confidence

这是整个产品演化中最重要的"减法"决策。

**删除时间**：v0.5 → v0.8 之间

**删除原因**（综合上述分析）：
1. AI 的 confidence 不等于统计学的置信区间
2. 同一项目两次评估的 confidence 波动 > 20%
3. 用户反馈"看到 confidence 忽高忽低，反而更不信任结果"
4. Benchmark Engine 接管了 confidence 的功能——用真实数据告诉用户"你在这个类别里排第几"

**代码痕迹**：在 `src/mock/valid-response.json` 和 `src/mock/contract-test.ts` 中仍然可以看到 `confidence` 和 `suitable_for` 的残留字段。这是有意保留的历史痕迹，证明了 schema 的演化过程。

**教训**：AI 产品中，"让 AI 对自己的输出表态"往往不如"用外部数据验证 AI 的输出"可靠。

### 为什么冻结功能

**冻结时间**：v1.5

**冻结原因**：
1. **功能完整性**：评估、评分、Benchmark、GitHub 分析、面试题、STAR 简历、开发周期、同类项目、批量对比、分享海报——核心决策链路已经闭环。
2. **Prompt 复杂度上限**：当前 Prompt 已经输出 20+ 字段。继续增加字段会导致：AI 输出不稳定、token 消耗过高、解析容错压力增大。
3. **维护成本**：每增加一个功能，parser.ts 就要增加一个解析函数，types/index.ts 就要增加一个接口。当前已有 11+ 解析函数。
4. **MVP 原则**：先验证核心价值，再迭代。详见 [[15 后续迭代路线图]]。

## 产品演化全景图

```
v0.1  AI评分工具（单分 + confidence）
  ↓ 删除 confidence
v0.3  AI评估工具（六维评分）
  ↓ 增加 Benchmark
v0.5  AI对比评估（Benchmark + 同类项目）
  ↓ 增加面试题、STAR简历、开发周期
v0.8  AI求职助手（完整评估报告）
  ↓ 增加 verdict、conclusion、compare
v1.0  AI项目决策助手（决策闭环）
  ↓ 功能冻结
v1.5  稳定版（当前版本）
```

## 优缺点分析

| 优点 | 缺点 |
|------|------|
| 每次转折都有明确的用户反馈驱动 | 没有提前做用户调研，走了弯路 |
| 删除 confidence 是正确的减法 | 删除时机偏晚（v0.5 才删） |
| 六维评分覆盖了核心决策维度 | 维度定义可能有争议（为什么没有"市场时机"？） |
| Benchmark 让分数有了参考系 | 基准数据是静态的，需要定期更新 |
| 功能冻结避免了过度膨胀 | 部分功能（如 effort.ts）开发后未使用 |

## 面试官可能追问

1. "为什么是六维而不是五维或七维？"
2. "删除 confidence 有没有做过 A/B 测试？"
3. "Benchmark 的数据来源是什么？怎么保证准确性？"
4. "如果重新做，你会怎么规划产品路线？"
5. "功能冻结的标准是什么？什么时候会解冻？"

## 标准回答

> **Q：如果重新做，你会怎么规划产品路线？**
>
> 如果重新来，我会在 Day 1 就确定"决策助手"而非"评分工具"的定位。核心区别是：评分工具输出一个数字，决策助手输出一个行动建议。我会直接跳过 confidence 字段，从六维评分 + verdict 开始，然后依次叠加 Benchmark（让分数有参考系）、面试题（覆盖求职场景）、批量对比（覆盖选择场景）。整个路线从 4 周缩短到 2 周。
