# 开发周期预测模块

> AI 预测 vs 规则引擎——为什么 AI 赢了

---

## 背景

Project Judge 的开发周期预测经历了两种方案：**规则引擎（effort.ts）** 和 **AI 直接预测**。最终 AI 方案胜出，规则引擎成为 dead code。这个决策过程本身就是一个值得记录的工程教训。

## 方案一：规则引擎（effort.ts）

### 设计思路

**文件**：`src/utils/effort.ts`

基于项目特征（页面数、API 数、是否有 AI、是否有云函数等）计算基准工期：

```typescript
const EFFORT_MAP = {
  pagePerUnit: 1.5,    // 每个页面 1.5 天
  apiPerUnit: 0.5,     // 每个 API 0.5 天
  aiSetup: 3,          // AI 能力接入 3 天
  cloudFn: 1.5,        // 云函数开发 1.5 天
  database: 2,         // 数据库设计+迁移 2 天
  auth: 3,             // 登录注册系统 3 天
  payment: 5,          // 支付集成 5 天
  testing: 2,          // 测试+修复 2 天
  deploy: 1,           // 部署上线 1 天
}
```

### 工作原理

1. `inferEffortInput(result)`：从 AI 评估结果中推断项目特征
   - 页面数 = `max(3, roadmap.length)`
   - API 数 = `max(1, floor(roadmap.length / 2))`
   - 是否有 AI = `true`（Project Judge 本身就是 AI 项目）
   - 是否有云函数 = 检查 techStack 是否包含 "cloud"/"lambda"/"function"
   - 是否有数据库 = 检查 techStack 是否包含 "sql"/"mongo"/"postgres"/"firebase"

2. `calculateBaseEffort(input)`：累加工时
   ```typescript
   let days = 0
   days += input.pageCount * 1.5
   days += input.apiCount * 0.5
   if (input.hasAI) days += 3
   // ... 以此类推
   return Math.ceil(days)
   ```

3. `daysToEstimatedTime(days)`：映射到枚举值
   - 1-3 天 → '1-3天'
   - 3-7 天 → '3-7天'
   - 7-14 天 → '1-2周'
   - 14-30 天 → '2-4周'
   - 30+ 天 → '1个月+'

### 为什么放弃

**问题 1：推断不准确**

`inferEffortInput` 从 AI 评估结果中推断项目特征，但这个推断本身就不准确：
- `pageCount = max(3, roadmap.length)`：roadmap 的长度不等于页面数
- `hasAI = true`：对所有项目都假设 true，明显不合理
- `hasCloudFn`：检查 techStack 是否包含 "cloud"，但很多项目用云函数但 techStack 里不会写

**问题 2：规则无法覆盖复杂场景**

一个"简单的 AI 聊天应用"和一个"复杂的 AI RAG 系统"都满足 `hasAI = true`，工时估算一样是 +3 天。但实际差异可能是 3 天 vs 30 天。

**问题 3：AI 直接预测更灵活**

当 Prompt V5 增加了 `estimated_dev_time` 和 `development_breakdown` 字段后，AI 可以：
- 基于项目复杂度给出合理的总工期
- 把工期拆成 5-8 个具体任务
- 每个任务有独立的天数

这比规则引擎的"累加工时"灵活得多。

### Dead Code 的现状

当前 `effort.ts` 仍然存在于代码库中，但**没有任何文件 import 它**。它是一个 dead code。

**为什么没删除？**
1. 作为历史记录保留，证明"我们考虑过规则引擎方案"
2. 未来可能有其他用途（如给用户提供"标准工时参考"）
3. 删除 dead code 的优先级低于新功能开发

## 方案二：AI 直接预测（最终方案）

### 设计

在 Prompt V5 中增加了两个字段：

```json
{
  "estimated_dev_time": "1-3天 或 3-7天 或 1-2周 或 2-4周 或 1个月+",
  "development_breakdown": [
    { "task": "需求分析", "days": 1 },
    { "task": "具体任务", "days": 2 }
  ]
}
```

### 为什么 AI 赢了

| 维度 | 规则引擎 | AI 预测 |
|------|---------|---------|
| 准确度 | 低（推断链条太长） | 中高（基于对项目的理解） |
| 灵活性 | 低（固定规则） | 高（每个项目独立评估） |
| 可解释性 | 高（每一步都可追溯） | 中（只有最终结果） |
| 维护成本 | 高（规则需要持续调优） | 低（Prompt 调整即可） |
| 覆盖范围 | 窄（只覆盖已知特征） | 宽（任何类型项目） |

**关键洞察**：开发周期预测本质上是一个**需要理解项目语义**的任务。规则引擎只能处理结构化特征（页面数、API 数），而 AI 可以理解"这个项目的难点在于实时音视频传输"这种非结构化信息。

### 教训

**规则引擎适合**：特征明确、规则稳定、不需要"理解"的场景（如 Benchmark 百分位计算）。

**AI 适合**：需要理解语义、场景多变、规则难以穷举的场景（如开发周期预测）。

**混合方案最佳**：Benchmark 用规则引擎（数据可靠），开发周期用 AI（灵活覆盖），verdict 用规则（`score >= 60`），其他用 AI。

## 优缺点分析

| 优点（AI 方案） | 缺点 |
|------|------|
| 灵活覆盖任何项目类型 | AI 的天数预测可能不准 |
| 不需要维护规则 | 没有可追溯的计算过程 |
| development_breakdown 让用户理解时间花在哪 | AI 可能低估复杂度 |
| 与评估结果自然衔接 | 每次更新 Prompt 可能影响预测 |

## 面试官可能追问

1. "effort.ts 还在代码里，为什么不删？"
2. "AI 预测的天数准确吗？"
3. "如果 AI 低估了开发周期怎么办？"
4. "规则引擎和 AI 可以结合使用吗？"

## 标准回答

> **Q：规则引擎和 AI 可以结合使用吗？**
>
> 可以，而且这可能是最优方案。具体做法是：用规则引擎计算"基准工时"（如页面数×1.5天），用 AI 预测"调整系数"（如"这个项目的实时通信模块比通常复杂 2 倍"）。最终工期 = 基准工时 × AI 调整系数。这样既有规则引擎的可解释性，又有 AI 的灵活性。当前没有实现是因为 MVP 阶段优先验证核心价值，复杂度优化留到 V2。
