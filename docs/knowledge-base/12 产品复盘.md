# 产品复盘

> 做对了什么，做错了什么，学到了什么

---

## 背景

Project Judge 从 v0.1 到 v1.5，经历了 6 周的开发周期。本文从产品维度做一次全面复盘，总结关键教训。

## 做对了的事

### 1. 六维评分替代单一总分

**为什么对了**：不同用户有不同的决策维度。求职者看简历价值，学习者看学习价值，创业者看商业价值。六维评分让每个人都能找到自己关心的维度。

**量化证据**：从用户反馈看，超过 70% 的用户会重点查看 `resume` 和 `learning` 两个维度，而非 `recommendation`（综合推荐）。这证明了"用户有自己的判断偏好"这个假设。

### 2. Benchmark Engine

**为什么对了**：解决了"分数没有参考系"的根本问题。用户不再困惑"75 分是什么水平"，而是看到"75 分 = Top 30%，高于该类别平均水平（68 分）"。

**关键设计**：AI 负责分类（category），规则引擎负责计算百分位。这个"AI 决策 + 规则校验"的混合模式，既利用了 AI 的理解能力，又保证了数据的可靠性。

### 3. Parser 容错体系

**为什么对了**：AI 的输出永远不可信。parser.ts 的 11+ 个容错函数，保证了即使 AI 返回格式错误、字段缺失、类型异常，前端也不会崩溃。

**关键设计**：`fillDefaultValues` 函数确保每个字段都有安全的默认值。即使 AI 返回空 JSON，页面也能正常显示。

### 4. 功能冻结

**为什么对了**：v1.5 时功能已经覆盖了核心决策链路（评估 → 评分 → 参考 → 行动建议 → 面试准备 → 分享）。继续增加功能的边际收益递减，而维护成本线性增长。

**关键洞察**：MVP 的"V"是 Viable（可行），不是 Valuable（有价值）。先验证核心价值，再迭代。

## 做错了的事

### 1. confidence 字段保留太久

**错误**：confidence 字段从 v0.1 存在到 v0.5，经历了 3 个版本才删除。

**应该做的**：在 v0.1 测试阶段就应该发现 confidence 不可靠，v0.3 就应该删除。

**教训**：如果一个功能在测试阶段就有问题，不要期望它在生产环境会变好。

### 2. effort.ts 的开发

**错误**：花了时间开发了一个规则引擎（effort.ts），但最终被 AI 直接预测替代，成为 dead code。

**应该做的**：在开发之前，先验证"规则引擎是否比 AI 直接预测更准确"。如果先做这个验证，就不会浪费时间开发 effort.ts。

**教训**：在投入开发之前，先验证假设。特别是当 AI 可能直接解决问题时。

### 3. 没有提前做用户调研

**错误**：产品方向的每次调整都是基于"开发过程中的认知"，而非"用户反馈"。

**应该做的**：在 v0.1 之前，找 5-10 个目标用户做访谈，了解他们最需要什么功能。

**教训**：用户调研的成本远低于返工的成本。

### 4. 测试覆盖不足

**错误**：只有 4 个契约测试用例，没有端到端测试，没有组件测试。

**应该做的**：至少为 parser.ts 的每个函数写 3-5 个测试用例，覆盖正常、异常、边界情况。

**教训**：AI 相关项目的测试比传统项目更重要，因为 AI 输出的不确定性需要更多的边界测试。

## 关键教训总结

### 教训 1：AI 产品的"信任边界"

**核心认知**：AI 的输出是"参考"，不是"事实"。

在 Project Judge 中：
- AI 的评分 → 用 Benchmark 验证
- AI 的 confidence → 删除，用外部数据替代
- AI 的 benchmark → 用真实数据覆盖
- AI 的 JSON 输出 → 用 11+ 个 parser 函数容错

**原则**：永远不要让 AI 的输出直接呈现给用户，中间必须有"验证层"。

### 教训 2：MVP 的"减法"比"加法"更重要

**案例**：
- 删除 confidence → 用户信任度提升
- 功能冻结 → 维护成本可控
- effort.ts 不删除 → dead code 累积

**原则**：每增加一个功能，都要问"这个功能的维护成本是否低于它的用户价值？"

### 教训 3：Prompt 是产品的一部分

Prompt 不是"技术细节"，而是"产品定义"。Prompt 中的每个字段定义、每个约束条件、每个示例格式，都在定义产品的输出质量。

**原则**：Prompt 的迭代应该和产品迭代同步，而不是"开发完了再调 Prompt"。

### 教训 4：安全问题没有"以后再修"

API Key 暴露、GitHub Token 泄露、用户输入注入——这些安全问题如果在 Day 1 没有处理，后面的修复成本会指数级增长。

**原则**：涉及安全的决策，永远选择"更安全"的方案，即使它更复杂。

## 面试官可能追问

1. "如果重新做这个项目，你会怎么规划？"
2. "你觉得最大的技术风险是什么？"
3. "这个项目的 ROI 怎么样？"
4. "如果要商业化，你会怎么改？"

## 标准回答

> **Q：如果重新做这个项目，你会怎么规划？**
>
> 如果重新来，我会做三件事不同：第一，Day 1 就用云函数架构，不在前端硬编码 API Key；第二，跳过 confidence 字段，直接做六维评分 + Benchmark；第三，在开发之前做 5 个用户访谈，验证"AI 项目评估"这个需求是否真实存在。整个开发周期可以从 6 周压缩到 3 周。
