# 工程踩坑记录

> 每一个坑都是真实踩过的，每一个修复都是验证过的

---

## 背景

Project Judge 的开发过程中踩了不少坑。这些坑有的是技术选型失误，有的是安全意识不足，有的是 AI 输出的不确定性。本文完整记录每个坑的发现过程、原因分析和修复方案。

## 坑 1：云开发误判

### 问题

项目初期，误以为"微信云开发"等同于"传统后端开发"，按照传统 Express/Koa 的思路设计云函数。

### 具体表现

- 试图在云函数中使用 `session` 管理状态（云函数是无状态的）
- 试图在云函数中做文件操作（云函数的文件系统是只读的）
- 试图在云函数中保持长连接（云函数执行完就销毁）

### 原因分析

微信云函数的运行模型是 **Serverless**，与传统后端有本质区别：
1. **无状态**：每次调用都是独立的，不保留上一次调用的状态
2. **短生命周期**：执行完就销毁，通常 60 秒超时
3. **只读文件系统**：只能读取打包时带入的文件
4. **冷启动**：首次调用或长时间未调用后，需要 1-2 秒初始化

### 修正方案

- 放弃 session，使用前端 `wx.setStorageSync` 管理状态
- 放弃文件操作，使用微信云数据库存储数据
- 放弃长连接，使用请求-响应模式
- 接受冷启动，前端加 loading 状态

### 教训

**在 Serverless 环境下，永远假设函数是"一次性的"。** 不要在云函数中维护任何运行时状态。

## 坑 2：API Key 暴露风险

### 问题

v0.1 版本中，DeepSeek API Key 直接硬编码在前端代码中。

### 发现过程

准备上线时，Claude 指出：微信小程序的代码会打包下发到用户手机。即使做了代码混淆，通过以下方式仍可提取 API Key：
1. **抓包**：使用 Charles/Fiddler 抓取 HTTPS 请求
2. **反编译**：使用 wxappUnpacker 反编译小程序包
3. **源码搜索**：在打包后的 JS 文件中搜索 `sk-` 前缀

### 影响

- 他人可以无限制调用 DeepSeek API，费用由你承担
- DeepSeek 可能因异常调用封禁你的 Key
- 无法控制调用频率和用量

### 修正方案

将 API Key 移到云函数的环境变量中：

```javascript
// cloud/evaluate/index.js
const apiKey = process.env.DEEPSEEK_API_KEY  // 从环境变量读取
```

在微信云开发控制台中配置环境变量，Key 永远不会出现在前端代码中。

### 教训

**涉及 API Key 的场景，Day 1 就应该用服务端中转。** 这不是可以"后面再改"的事情。

## 坑 3：confidence 设计失误

### 问题

v0.1 中设计了 `confidence` 字段（0-1），让 AI 告诉用户"我对这个评分有多大把握"。

### 实际表现

同一项目输入两次：
- 第一次：confidence = 0.85
- 第二次：confidence = 0.62

波动超过 20%，用户看到后反而更不信任结果。

### 根本原因

LLM 的 confidence 不等于统计学的置信区间。它是"语气的确定程度"，而非"准确概率"。AI 可以对一个完全错误的评估给出 confidence = 0.95，因为它"觉得自己很确定"。

### 修正方案

1. 删除 confidence 字段
2. 引入 Benchmark Engine，用真实数据告诉用户"你在这个类别里排第几"
3. 用 `verdict`（recommend / not_recommend）替代模糊的置信度

### 教训

**AI 产品中，"让 AI 对自己的输出表态"不如"用外部数据验证 AI 的输出"可靠。** 如果需要告诉用户"这个结果有多可信"，用客观数据而非 AI 的主观判断。

## 坑 4：Prompt 返回字段不足

### 问题

Prompt V1 只要求输出 6 个字段（project_name, score, confidence, verdict, pros, cons）。随着产品演化，需要的字段增加到 20+，但 Prompt 没有同步更新。

### 具体表现

- 增加了面试题功能，但 Prompt 中没有 `interview_questions` 字段定义
- AI 返回的 JSON 中没有这个字段，前端显示空白
- 用户反馈"面试题功能在哪里？"

### 修正方案

每次增加新功能时，同步更新：
1. Prompt 模板（cloud/evaluate/index.js 中的 `getSystemPrompt`）
2. TypeScript 接口（src/types/index.ts 中的 `AIResponse`）
3. 解析函数（src/services/parser.ts 中的对应 parse 函数）
4. 默认值填充（`fillDefaultValues` 函数）

### 教训

**Prompt 和代码之间存在"隐式契约"。** Prompt 定义了 AI 输出什么，代码定义了前端期望什么。两者必须同步更新，否则会出现字段缺失。

## 坑 5：Parser 容错设计

### 问题

早期版本的 parser 使用一个大的 try-catch 包裹所有解析逻辑。一旦某个字段解析失败，整个结果丢失。

### 具体表现

AI 返回的 JSON 中，`scores.resume.score` 突然变成了字符串 `"85"`（而不是数字 85）。JSON.parse 成功，但后续的数值比较失败，整个页面白屏。

### 修正方案

采用**"每个字段一个解析函数"**的设计模式：

```typescript
// clampScore：将任何值转为 0-100 的数字
export function clampScore(score: unknown): number {
  const n = Number(score)
  if (!Number.isFinite(n)) return 0
  return Math.max(0, Math.min(100, Math.round(n)))
}
```

每个解析函数独立处理异常：
- `clampScore`：非数字 → 0，超出范围 → 钳位到 0-100
- `normalizeVerdict`：非 recommend → not_recommend
- `parseSimilarProjects`：非数组 → 空数组
- `parseBenchmark`：非对象 → 默认值 { average: 65, percentile: 'Top 50%' }

### 关键设计：fillDefaultValues

```typescript
export function fillDefaultValues(partial: Partial<EvaluationResult>): EvaluationResult {
  return {
    projectName: ensureString(partial.projectName, '未命名项目'),
    summary: ensureString(partial.summary, '暂无'),
    verdict: partial.verdict ?? 'not_recommend',
    // ... 每个字段都有默认值
  }
}
```

**即使 AI 返回了一个完全空的 JSON，前端也能正常显示（只是所有字段都是默认值）。**

### 教训

**LLM 的输出永远不可信。** 即使你要求了 JSON 格式、给了 Schema、设了 temperature=0，AI 仍可能返回：
- 类型错误（字符串 vs 数字）
- 范围错误（score = 999）
- 字段缺失（整个字段不存在）
- 完全无关的内容

**Parser 的设计原则：永远不崩溃，永远有默认值。**

## 坑 6：GitHub API 读取限制

### 问题

未认证的 GitHub API 限制为 60 次/小时/IP。在演示场景中，连续评估 10+ 个 GitHub 项目后，API 返回 403。

### 具体表现

```
GitHub API rate limit exceeded
X-RateLimit-Limit: 60
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1718361600
```

### 修正方案

**短期**：云函数中 catch GitHub API 错误，返回 null，评估继续（只是没有真实数据）

```javascript
try {
  repoInfo = await fetchGitHubRepo(content.trim())
} catch {
  repoInfo = null  // 评估继续，只是没有 GitHub 数据
}
```

**长期**：
1. 申请 GitHub Personal Access Token，额度提升到 5000 次/小时
2. 引入缓存层，同一仓库 24 小时内不重复请求
3. 在前端提示用户"GitHub 数据获取失败，请稍后重试"

### 教训

**调用第三方 API 时，永远要考虑限流。** 做好降级方案：即使 API 失败，核心功能也不能中断。

## 坑 7：extractJson 问题

### 问题

DeepSeek 的输出不总是纯 JSON。有时会包含：
- Markdown 代码块包裹：```` ```json {...} ``` ````
- 前后夹杂文本："以下是评估结果：\n{...}\n以上是评估结果"
- JSON 内部有注释：`{ "key": "value" // 注释 }`

### 早期版本

```javascript
function extractJson(text) {
  return JSON.parse(text)  // 直接解析，失败就崩溃
}
```

### 最终版本

**文件**：`cloud/evaluate/index.js:257-269`

```javascript
function extractJson(text) {
  const cleaned = text.trim()
    .replace(/^```(?:json)?\s*/i, '')  // 去掉开头的 ```json
    .replace(/\s*```$/i, '')           // 去掉结尾的 ```

  try { return JSON.parse(cleaned) } catch (_) {}

  // Fallback：提取第一个 { 到最后一个 } 之间的内容
  const start = cleaned.indexOf('{')
  const end = cleaned.lastIndexOf('}')
  if (start === -1 || end === -1) {
    throw new Error('未找到有效 JSON')
  }
  return JSON.parse(cleaned.slice(start, end + 1))
}
```

**三层容错**：
1. 直接解析（处理纯 JSON）
2. 去掉 Markdown 包裹后解析
3. 提取 `{` 到 `}` 之间的内容后解析

### 教训

**LLM 的输出格式永远是薛定谔的 JSON。** 你要求 JSON，它可能给你 Markdown 包裹的 JSON；你要求纯 JSON，它可能给你前后有文本的 JSON。extractJson 必须处理所有这些情况。
