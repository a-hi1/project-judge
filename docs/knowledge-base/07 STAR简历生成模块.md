# STAR 简历生成模块

> 让 AI 帮用户写出"可直接写入简历"的项目描述

---

## 背景

STAR（Situation-Task-Action-Result）是面试和简历写作中最常用的结构化表达框架。Project Judge 自动为每个评估项目生成 STAR 格式的简历描述，解决了一个普遍痛点：**很多人做了不错的项目，但写不出好的简历描述**。

## 为什么这样设计

### 用户痛点

典型的简历写法：
> "负责开发了 AI 项目评估系统"

这种写法的问题：
1. 没有背景（为什么做？）
2. 没有具体技术（怎么做的？）
3. 没有量化成果（做到什么程度？）
4. 面试官看到后无法追问

### 期望的写法

> "独立开发 AI 项目评估系统，基于 Taro + React + TypeScript 构建微信小程序前端，集成 DeepSeek API 实现五维智能评分。设计 Prompt 分层路由策略（GitHub/idea/description 三种输入场景），实现结构化 JSON 解析容错机制。累计支持 100+ 项目评估，平均生成时间 15 秒内，结果解析成功率 98%。"

### 设计目标

1. **零编辑**：生成的内容可以直接复制粘贴到简历
2. **有数据**：必须包含量化成果
3. **有技术**：必须提到具体技术方案
4. **有深度**：不是"负责开发"，而是"设计了什么、解决了什么问题"

## 实现细节

### Prompt 设计

```json
{
  "resume_star": {
    "situation": "项目背景，30字以内",
    "task": "你的任务，20字以内",
    "action": "你做了什么，必须包含具体技术方案，30字以内",
    "result": "成果，必须包含量化数据，如提升XX%、支持XX+、降低XX%，20字以内",
    "resumeText": "可直接写入简历的描述，80-120字，必须包含量化成果"
  }
}
```

**关键约束**：

| 字段 | 约束 | 原因 |
|------|------|------|
| situation | 30 字以内 | 简历空间有限，背景要精简 |
| task | 20 字以内 | 任务描述要一句话说清 |
| action | 必须包含具体技术方案 | 防止"负责开发"这种空话 |
| result | 必须包含量化数据 | 面试官最看重的结果 |
| resumeText | 80-120 字 | 简历一行的合理长度 |

### Prompt 中的负面约束

```
action 必须包含具体技术方案，禁止"负责开发"、"实现功能"这种空话
result 必须包含量化数据，例如"累计评估 100+ 项目"、"平均响应时间 15 秒"、"Prompt 解析成功率 98%"
```

**为什么用负面示例？** 因为正面约束（"要有具体技术"）太抽象，AI 会钻空子。负面约束（"禁止空话"）更具体，AI 知道什么不能写。

### 示例格式的作用

Prompt 中给出了完整的示例：

```
resumeText 示例："独立开发 AI 项目评估系统，基于 Taro + React + TypeScript 构建微信小程序前端，
集成 DeepSeek API 实现五维智能评分。设计 Prompt 分层路由策略（GitHub/idea/description 三种输入场景），
实现结构化 JSON 解析容错机制。累计支持 100+ 项目评估，平均生成时间 15 秒内，结果解析成功率 98%。"
```

这个示例不只是告诉 AI "格式是什么"，更是告诉它"质量标准是什么"。AI 会自动模仿这个示例的结构和深度。

### 前端解析

**文件**：`src/services/parser.ts:100-112`

```typescript
function parseResumeStar(val: unknown): ResumeStar {
  if (!val || typeof val !== 'object') {
    return { situation: '', task: '', action: '', result: '', resumeText: '' }
  }
  const s = val as Record<string, unknown>
  return {
    situation: ensureString(s.situation),
    task: ensureString(s.task),
    action: ensureString(s.action),
    result: ensureString(s.result),
    resumeText: ensureString(s.resumeText),
  }
}
```

**容错设计**：如果 AI 没有输出 `resume_star` 字段，返回空对象而非报错。前端会显示"暂无简历建议"。

## 备选方案

### 方案 A：只输出 resumeText，不输出 STAR 分解

- 优点：Prompt 更简洁
- 缺点：用户无法理解和修改各个部分
- 结论：需要分解，让用户理解每个部分的逻辑

### 方案 B：让用户输入自己的项目信息，AI 基于输入生成

- 优点：更个性化
- 缺点：增加用户输入负担
- 结论：当前方案已经基于评估结果生成，足够个性化

## 优缺点分析

| 优点 | 缺点 |
|------|------|
| 零编辑，可直接写入简历 | AI 生成的量化数据可能是编造的 |
| 负面约束 + 示例格式保证质量 | 不同项目生成的 action 可能雷同 |
| STAR 分解让用户理解每个部分 | 需要用户自己调整措辞以匹配个人经历 |
| 80-120 字限制保证简历格式 | 对于特别复杂的项目，120 字可能不够 |

## 面试官可能追问

1. "AI 生成的量化数据是真实的吗？"
2. "怎么保证生成的简历描述不千篇一律？"
3. "STAR 框架是万能的吗？有没有它不适用的场景？"

## 标准回答

> **Q：AI 生成的量化数据是真实的吗？**
>
> 不完全真实。AI 给出的"累计评估 100+ 项目"是基于项目能力推测的合理数据，而非真实统计。这是当前方案的一个已知局限。解决方向有两个：1）接入真实的使用统计数据（需要后端存储），把真实数据注入 Prompt；2）在前端提示用户"请将量化数据替换为你的真实数据"。目前采用方案 2，在 UI 中加了提示。
