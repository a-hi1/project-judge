# 简历写法

> 如何在简历中展示 Project Judge 项目

---

## 背景

Project Judge 本身就是一个帮用户写简历的工具。但反过来，**如何在简历中写 Project Judge 这个项目**，同样需要技巧。本文提供具体的简历写法建议。

## STAR 格式写法

### 通用版本

> **Project Judge — AI 项目决策助手**
>
> **Situation**：独立开发 AI 项目评估系统，解决"项目想法是否值得投入"的决策痛点。
>
> **Task**：设计并实现一个基于大语言模型的多维度项目评估工具，覆盖评分、Benchmark、面试准备、简历生成全链路。
>
> **Action**：
> - 基于 Taro + React + TypeScript 构建微信小程序前端，设计六维评分体系（简历/学习/商业/难度/创新/推荐）
> - 集成 DeepSeek API 实现智能评估，设计 Prompt 分层路由策略（GitHub/idea/description 三种输入场景）
> - 开发 Benchmark Engine，基于 15 类项目基准数据计算百分位排名，让分数有参考系
> - 实现 11+ 结构化 JSON 解析容错函数，保证 AI 输出的稳定性和可靠性
> - 集成 GitHub REST API，5 路并行获取仓库元数据、README、技术栈、贡献者、发布信息
>
> **Result**：
> - 支持 100+ 项目评估，平均响应时间 15 秒内
> - JSON 解析成功率 98%，六维评分覆盖率 100%
> - 生成可直接写入简历的 STAR 格式描述 + 5 题面试准备材料

### 针对不同岗位的变体

**前端岗位**：
> 基于 Taro + React + TypeScript 构建微信小程序，设计暗色主题设计系统（参考 Linear/Raycast），实现 Canvas 2D 海报生成、状态机模式评估流程、双层类型系统（snake_case ↔ camelCase）。

**后端/AI 岗位**：
> 设计微信云函数中转架构，集成 DeepSeek API（temperature=0.3, json_object 格式），开发 Prompt 分层路由策略，实现 Benchmark Engine（15 类项目 × 4 百分位阈值），集成 GitHub REST API 5 路并行调用。

**全栈岗位**：
> 独立完成从需求分析到上线的全流程：前端（Taro/React/TS）、后端（微信云函数）、AI（DeepSeek Prompt 工程）、数据（GitHub API + Benchmark）、测试（契约测试 + 容错验证）。

## 技术亮点提炼

### 亮点 1：Prompt 分层路由

**简历描述**：
> 设计 Prompt 分层路由策略，根据输入类型（GitHub URL / 详细描述 / 一句话想法）选择不同的评估模板，GitHub 场景注入真实仓库数据（Stars、技术栈、README）。

**面试展开**：
- 为什么分层？不同输入的信息量不同，需要不同深度的评估
- 怎么实现？`getSystemPrompt(type, repoInfo)` 函数根据 type 选择 intro
- 效果如何？GitHub 项目的评估准确度明显高于 idea 项目

### 亮点 2：Benchmark Engine

**简历描述**：
> 开发 Benchmark Engine，基于 15 类项目基准数据（AI Agent / RAG / SaaS 等）计算百分位排名，将 AI 评分从"绝对分数"升级为"相对排名"。

**面试展开**：
- 为什么需要？AI 的绝对分数没有参考系
- 怎么实现？AI 输出 category + recommendation.score，规则引擎计算百分位
- 关键设计：不信任 AI 的 benchmark 输出，用真实数据覆盖

### 亮点 3：Parser 容错体系

**简历描述**：
> 实现 11+ 结构化 JSON 解析容错函数，覆盖类型异常（字符串/数字混淆）、范围异常（score=999）、字段缺失、格式异常（Markdown 包裹 JSON）等场景，解析成功率 98%。

**面试展开**：
- 为什么需要？AI 输出永远不可信
- 怎么实现？每个字段一个解析函数 + `fillDefaultValues` 兜底
- 关键设计：三层 extractJson 容错（纯 JSON → 去 Markdown → 提取花括号）

### 亮点 4：GitHub 深度分析

**简历描述**：
> 集成 GitHub REST API，5 路并行获取仓库元数据、README、package.json 技术栈、贡献者数量（Link header 解析）、发布信息，自动计算活跃度/社区/影响力/风险四维指标。

**面试展开**：
- 为什么 5 路并行？减少用户等待时间
- 贡献者数量怎么获取？从 Link header 的 `rel="last"` page 值推算
- 技术栈怎么提取？从 package.json 的 dependencies 中过滤 40+ 已知框架

### 亮点 5：设计系统

**简历描述**：
> 设计 Linear/Raycast 风格暗色主题系统，定义三层背景色、8rpx 基础间距、9 级字号体系、5 级圆角，封装复用样式对象（cardStyle/pillStyle/dividerStyle）。

## 量化数据建议

简历中尽量包含量化数据。以下数据可以从项目中提取：

| 数据 | 来源 | 说明 |
|------|------|------|
| 100+ 项目评估 | 历史记录 | 累计评估数 |
| 15 秒平均响应 | 云函数日志 | DeepSeek + GitHub 总耗时 |
| 98% 解析成功率 | 契约测试 | 4 个测试场景全部通过 |
| 20+ 输出字段 | Prompt 设计 | AI 返回的字段数 |
| 15 类基准数据 | benchmark.json | Benchmark 覆盖的类别数 |
| 11+ 容错函数 | parser.ts | 解析函数数量 |
| 5 路并行 API | 云函数代码 | GitHub API 调用数 |
| 6 维评分体系 | types/index.ts | 评分维度数 |
| 1080px 海报 | poster.ts | 海报分辨率 |

## 面试官可能追问

1. "这个项目有多少用户？"
2. "你一个人做的？有没有遇到什么困难？"
3. "这个项目的技术难点在哪里？"
4. "如果要扩展到 10 万用户，你怎么改？"

## 标准回答

> **Q：这个项目的技术难点在哪里？**
>
> 三个核心难点。第一，Prompt 工程：如何让 AI 稳定输出 20+ 个结构化字段，同时保证每个字段的质量。我通过分层路由、示例格式、负面约束、字数限制四层策略解决。第二，输出容错：AI 的 JSON 输出随时可能格式错误、字段缺失、类型异常。我设计了 11+ 个独立的解析函数，每个字段单独容错，即使 AI 返回空 JSON 也能正常显示。第三，数据可信度：AI 的评分没有参考系，我引入了 Benchmark Engine，用 15 类项目的真实基准数据计算百分位排名。
